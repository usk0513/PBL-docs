# なぜGitを使うのか？

<!-- {"layout": "タイトルと左半分テキスト・右半分画像"} -->

## なぜGitを使うのか？

### 例1: 実装の巻き戻し失敗

- 新しい機能を追加したが動かなかった
- 一旦元に戻そうと思い、追加部分を手動で削除
- しかし、プログラムが動かない...
- **何を変更したか正確には覚えていない**

---

<!-- {"layout": "タイトルと左半分テキスト・右半分画像"} -->

## なぜGitを使うのか？

### 例2: 複数人での同時作業の煩雑さ

- 複数人で同じファイル（例: `本番資料.pptx`）を作成する場合
- 各自がファイルをコピー
  - `【廣野作業用】本番資料.pptx`
  - `本番資料_小浜.pptx`
- 最後は1つへまとめる作業が面倒
- さらに問題:
  - コピー元のファイルをどの時点で取得したかによって各自の作業内容が変わる
  - 最終的にまとめた資料では不要なスライドが残る
  - 他の人が修正した内容は上書きされて消える

---

<!-- {"layout": "タイトルと左半分テキスト・右半分画像"} -->

## なぜGitを使うのか？

### 例3: 変更履歴の追跡困難

- 「誰が、いつ、なぜこのコードを変更したのか？」が分からない
- そのため、バグの原因特定が困難になる

### バックアップの課題

- `プロジェクト_最終版.pptx`
- `プロジェクト_最終版2.pptx`
- `プロジェクト_本当に最終版.pptx`

→ **どれが本当の最終版？**

## Gitとは

---
<!-- {"layout": "タイトルと本文"} -->
## Gitとは
- バージョン管理を実現するソフトウェア
- ソースコードの変更履歴を管理
- 複数人(チーム)での並行同時作業を支援
- 無料で利用可能
- オープンソースで広く利用されており、事実上のデファクトスタンダード

---
<!-- {"layout": "タイトルと本文"} -->
## GitHubとは
- Gitで管理されたソースコードをインターネット上でホスティングするサービス
- チームでの共同作業を容易にする機能を提供
  - Pull Request、Issue管理、GitHub Actionsなど
- Gitと連携してソースコードのバージョン管理を実現
- 多くのオープンソースプロジェクトがGitHub上でホストされている
- GitHubはMicrosoftの一部門でありGitHub CopilotなどのAI支援ツールも提供

## PBLでのソースコード管理の流れ： ステップ1
- ベースとなるソースコードはGitHub上にある
  - Gitで管理されているひとまとまりのソースコードの保管場所を「リポジトリ」と呼ぶ
  - 特にGitHubのようなインターネット上のリポジトリを「リモートリポジトリ」と呼ぶ
- ソースコードを自分のPCで編集するために、まずソースコードを丸ごとローカルに複製する
  - この操作を「クローン」と呼ぶ
  - ローカルにあるリポジトリを「ローカルリポジトリ」と呼ぶ

## PBLでのソースコード管理の流れ： ステップ2

## PBLでのソースコード管理の流れ： ステップ3
- 作成したブランチで作業する

## PBLでのソースコード管理の流れ： ステップ4
- 編集内容を記録する
  - 区切りの良いタイミングで変更を保存
  - 変更内容が保存されるのは作業しているブランチ
- 記録するために2つの操作が必要
  1. 変更を記録するファイルを選択する
     - 選択されたファイルは「ステージングエリア」に追加される
     - この操作を「アド」と呼ぶ
  2. 選択したファイルの変更をローカルリポジトリに保存する
     - この操作を「コミット」と呼ぶ
     - コミットするときにはどんな変更をしたかメッセージを付ける

  - リモートリポジトリから最新の変更を取得する操作を「プル」と呼ぶ

## ステップ1: git clone

### GitHubからソースコードを取得

1. GitHubリポジトリページで緑の「Code」ボタンをクリック
2. HTTPSタブを選択
3. URLをコピー
4. ターミナルで`git clone`コマンドを実行

```bash
git clone https://github.com/example/project.git
```

**注**: 具体的なURLは別途手順書を参照

# ステップ2: ローカルで新しい作業場を作る

git checkout -b

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ2: git checkout -b

### 新しいブランチを作成

```bash
git checkout -b <ブランチ名>
```

### ブランチ命名規則

- `チーム名/機能名` の形式を推奨
- 例: `team-a/add-login-feature`

```bash
git checkout -b team-a/add-feature
```

**補足**: mainブランチから新しいブランチを作成し、そのブランチに切り替える

# ステップ4: 編集内容を記録する（前半）

git add

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ4（前半）: git add

### ステージングに追加

```bash
git add <ファイル名>
```

または

```bash
git add .
```

### ステージングエリアとは

- コミットする変更を準備する場所
- `git add`で変更をステージングエリアに追加
- 次のステップでステージングされた変更をコミットする

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ4（前半）: git add の例

### 特定のファイルを追加

```bash

```

### 全ての変更を追加

```bash
git add .
```

# ステップ4: 編集内容を記録する（後半）

git commit

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ4（後半）: git commit

### コミットで変更を記録

```bash
git commit -m "コミットメッセージ"
```

### コミットメッセージの書き方

- 何を変更したかを分かりやすく書く
- 例: 「ログイン機能を追加」「バグを修正」「READMEを更新」
- フォーマットの厳密な規定はなし（チームで決める）

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ4（後半）: git commit の例

```bash
git commit -m "ログイン機能を追加"
```

**補足**: コミットすることで、変更がローカルリポジトリの履歴に記録される

# ステップ5: クラウド側にも編集内容を同期

git push

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ5: git push

### リモートに変更を送信

```bash
git push origin <ブランチ名>
```

### 例

```bash
git push origin team-a/add-feature
```

**補足**:

- ローカルでコミットした変更をGitHub（リモート）に送信
- これでチームメンバーが変更を見られるようになる

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ5: git push - 認証について

### HTTPS接続の場合

- ユーザー名とパスワードを求められる場合がある
- 必要に応じてPersonal Access Token（PAT）を使用
- 詳細は別途手順書を参照

# ステップ6: マスターデータに作業内容を統合

PR・マージ

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ6: PR・マージ

### Pull Request（PR）とは

- 自分の変更をmainブランチに取り込んでもらうための依頼
- チームメンバーがコードをレビューできる

### 手順

1. プッシュ後、GitHub上で「Compare & pull request」ボタンが表示される
2. ボタンをクリックしてPR作成画面を開く
3. PRのタイトルと説明を入力
4. 「Create pull request」ボタンをクリック
5. レビュー後、「Merge pull request」ボタンでマージ

---

<!-- {"layout": "タイトルと本文"} -->

## ステップ6: PR・マージ - 補足

**補足**:

- レビュー・マージのルールはチームで決める
- マージ後、変更がmainブランチに反映される
